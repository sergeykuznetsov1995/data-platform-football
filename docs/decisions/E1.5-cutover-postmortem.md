## Архитектурные решения (подтверждены кодом)

- **D1** — **Inline `'fb_' || player_id` prefix** (Вариант A) для dim_player + fct_player_match вместо JOIN на silver.xref_player. Rationale: silver.xref_player canonical_id для FBref-source гарантированно равен `'fb_' || player_id` (см. xref_player_resolver.py + xref_player.sql validation regex `^fb_`). JOIN был бы redundant + дополнительная нагрузка на Trino.

- **D2** — **Season-predicate в каждом silver.xref_team JOIN**. Per CLAUDE.md / `feedback_xref_join_season_predicate.md`: silver.xref_team имеет per-(source, source_id, league, season) rows; без AND по league+season fan-out 1.5-4×. Все 5 переписанных JOIN'ов (dim_match × 2, fct_player_match, match_outcomes × 2, dim_standings) включают `AND xt.league = m.league AND xt.season = CAST(m.season AS varchar)`.

- **D3** — **CAST(season AS varchar) в JOIN-side**. silver.xref_team.season — varchar; FBref Bronze хранит bigint year-of-start. Унификация на JOIN-side (`CAST(m.season AS varchar)`) — silver таблица остаётся как есть, downstream таблицы остаются с bigint. Только predicate приводит типы.

- **D4** — **Type contract preservation**: dim_team.season остаётся BIGINT через `CAST(season AS bigint)` в финальном SELECT, чтобы downstream (mart_event_heatmap, fct_player_unavailable JOIN на dim_match) не сломались.

- **D5** — **Один prep-PR, один execute-merge, один follow-up cleanup PR** (3 PR'а). Rationale: minimal-blast-radius pattern. Prep-PR не меняет prod data (silver.xref_* уже стабильны, refactor только чтобы Gold DAG читал silver). Execute-merge запускает Gold DAG с новой формулой → first run переписывает все cascade tables. Follow-up drop'ает entity_xref после ≥1 successful Gold run.

- **D6** — **Severity=WARNING для всех 6 новых ref_integrity check'ов в prep-PR**. Rationale: gate-watch окно (3 дня) — наблюдаем drift через Telegram без падения DAG'а. Promotion в ERROR — отдельный backlog item после ≥3 дня green DQ post-cutover.

- **D7** — **diff_threshold=100 default сохранён в prep-PR**, TODO в xref_dq.py docstring снизить до 10 после cutover. Rationale: текущий baseline cid_diff=43 → новый threshold 10 спамил бы Telegram во время gate-watch.

- **D8** — **Magic-number `_PARITY_BASELINE_TEAM_CID_DIFF = 43`** в xref_dq.py с inline-comment ссылкой на `data/audit/e1_parity_2026-05-09.json` + `docs/decisions/E1.5-cutover-prep.md`. Rationale: явная связь с baseline JSON чтобы при следующем drift-update легко обновить.

- **D9** — **Subagent-driven параллелизм окупился снова**: 7 subagent'ов в 3 волны (T0+T1 → T2+T3+T4 → T5 → T6 → T7) сжали 1-2 дня roadmap-плана в <1 часа wall-clock. Coordination overhead — нулевой (каждый агент работал в независимом file-set). Pattern E1 переиспользован.
