## 1. Executive summary

E1 (closed 2026-05-08) материализовал `iceberg.silver.xref_{team,match,referee,manager,player}` и оставил их dual-run рядом с легаси `iceberg.gold.entity_xref`. E3/E4 факты (8 fact-таблиц, перечисленных в parity JSON `gold_facts_using_silver_xref`) уже читают silver. Но 6 файлов Gold-слоя остались на легаси:

* `dim_team`, `dim_match`, `dim_player`, `dim_standings`, `fct_player_match`, `match_outcomes`

Это создаёт **canonical_id drift** внутри Gold: одни таблицы пишут `manchester_utd` (legacy regex), другие `manchester_united` (silver alias-canonical). Baseline JSON фиксирует это как **43 cid_diff** на team-xref (78.5% canonical_id_match_pct vs DoD 100%) и потенциальные orphan FK-связи между новыми и старыми ветками графа Gold.

**E1.5 cutover** перенаправляет 6 файлов на `silver.xref_*` и снимает блок-риск ref-integrity (E3/E4 facts ↔ dim_team/dim_match). Сам merge ждёт открытия gate'а; этот документ описывает **prep**: что меняется, какие riski, как откатываться, какой checklist на каждый день dual-run.

**Bлок-риски снимаются**:
1. Расхождение canonical (`manchester_utd` ≠ `manchester_united`) между E3/E4 facts и dim_*.
2. Невозможность ref-integrity DQ check на `fct_event.team_id_canonical → dim_team.team_id` (отключено в E3 DQ как `match_id_canonical='whoscored_raw'` v0_unbridged).
3. Дубликат source-of-truth для cross-source resolution (legacy emit'ит только FBref, silver — все 8 источников).

## 2. Alias-drift register

### 2.1 Формула drift

Legacy `gold.entity_xref.sql` (lines 22-26):

```sql
-- direct regex normalisation, no alias lookup
LOWER(REGEXP_REPLACE(team_name, '[^a-zA-Z0-9]+', '_'))
```

Silver `xref_team.sql.j2` использует ту же regex-формулу, **но** перед ней резолвит `team_name` через `configs/medallion/team_aliases.yaml`:

```
raw_name -> alias_lookup(yaml) -> canonical_name -> regex_canonical
```

Если `raw_name` найден в `aliases._generic` или `aliases.<source>` любого блока — берётся `canonical_name` (Wikipedia-official). Если не найден — fallback на тот же regex (тогда legacy == silver).

## 5. Gate-watch process (daily checklist)

**Цель**: 3 successive green-parity DAG runs `dag_transform_xref` подтверждают cutover-ready.

**Cadence**: один раз в сутки после daily master-pipeline (14:00 UTC), check выполняется в окне 14:30-16:00 UTC.

### 5.1 Команда — pull parity_summary из XCom

```bash
# Найти последний run_id для dag_transform_xref
docker compose exec airflow-webserver airflow dags list-runs \
    -d dag_transform_xref --no-backfill --state success \
    | head -3

# Pull parity_summary XCom
docker compose exec airflow-webserver airflow tasks render \
    dag_transform_xref validate_xref <RUN_ID>

# Альтернатива — Airflow UI: DAG → validate_xref → XCom tab → key=parity_summary
```

### 5.2 Что мониторить (3 KPI per day)

| Metric | Path в parity_summary | Green criteria | WARNING | RED (block gate) |
|---|---|---|---|---|
| Team `gold_only` | `team.gold_only` | `= 0` | `> 0`, but explainable (FBref Bronze dropped row) | `> 0` без explainable trigger — silver SQL regression |
| Team `cid_diff` | `team.cid_diff` | `43 ± 5` (stable around baseline) | `> 50` — расширение alias drift, нужно triage | `< 38` или `> 60` — структурный сдвиг |
| Match `gold_only` | `match.gold_only` | `= 0` | `> 0` | `> 0` consistent across days |
| Match `cid_match_pct` | `match.canonical_id_match_pct` | `= 1.0` | — | `< 1.0` consistent |

Telegram-watcher уже настроен (`xref_dq.maybe_alert_parity` шлёт при `total diff ≥ 100`); T4 (`data-quality-agent` deliverable) расширит threshold detection с per-entity granularity.

### 5.3 Gate criteria (для PR-2 execute)

Cutover открыт если **3 successive daily** runs показали:

1. ✅ `team.gold_only = 0` для FBref-only subset.
2. ✅ `team.cid_diff` в диапазоне `38-48` (stable around 43 baseline).
3. ✅ `match.gold_only = 0` AND `match.cid_match_pct = 1.0`.
4. ✅ Player rejection rate stable (ниже R8 fix не блокер; `understat ≤ 20%`, `whoscored ≤ 20%` — known limitation).
5. ✅ No Telegram WARNING/ERROR alerts от `xref_dq` validators за 72 часа.

Earliest gate open: **2026-05-11**. Realistic: **2026-05-12**.

### 5.4 Daily log template

Создать `data/audit/e1_5_gate_watch_<date>.md` со снапшотом parity_summary + verdict (PASS/WARN/BLOCK):

```
2026-05-09  baseline: team.gold_only=0  team.cid_diff=43  match=100%  PASS (day 0)
2026-05-10  TBD
2026-05-11  TBD  <-- earliest gate open if 3-of-3 PASS
```

## 6. Rollback runbook

### Plan-A — Revert PR (preferred)

Триггер: ref-integrity DQ failure post-execute, или Superset/dashboards показывают NULL `team_id`.

1. `git revert <PR-2 merge commit>` на ветке `master`.
2. Triggered Gold DAG автоматически перезапустится с легаси-`gold.entity_xref` references — **dual-run всё ещё работает**, downstream запросы вернутся к `manchester_utd` форме.
3. Notify Telegram + log in `data/audit/`.

**RTO**: 15 минут (один Gold DAG run ~12 минут).
